The small story to begin with

Once upon a time there was a cool C++ toolkit for developing cross-platform applications called Qt. But there was no cross-platform IDE suitable for creating cross-platform applications with it. "That is unfair!", thought young brave developer FullMetalCoder and initiated one. His child name was DevQt...
Time passed. DevQt has been growing up. Developers liked it. Some joined to help FullMetalCoder raising his child. But each of them had his own opinion about the way DevQt should behave. So he did his best to implement this or that feature. This was a road to nowhere...
Now the time has come to make everything organized! The time has come to make separated developers to start acting as a team!

Being responsible

For the first sight it seems participating in an Open Source project bares no responsibilities. You could join some project "just for fun". But in case if you joined it for the others benefit you should think about it once again.
If something goes wrong there's nobody to blame for it, because nobody's responsible for it. On the other hand, if the project is successful who's success is it? Nobody's! Because nobody is responsible for it.

Localization

Any modern piece of software should be able to "speak" to user his own language. DevQt is not an exception.
Tasks:

  • keeping DevQt translation-aware
  • involving translators
  • monitoring translators' work
Responsible: Cesar

Deployment

Prebuilt binary packages help to save time compiling DevQt from sources and solving problems with integration into existing system. These packages also help those, who just started their programmer career and are unfamiliar with all the command-line compiling stuff. Needless to say this piece of work is important too.
Tasks: Depend on chosen deployment packages.
Responsible: Cesar, volunteers needed

Managing patches

One of the benefits of Open Source is that anybody can participate. There're different ways of doing so. One of them is sending patches and bug-fixes.
Tasks:

  • keeping track of patch submissions
  • reviewing submissions
  • committing or rejecting straightforwards patches
  • assigning patches to other coders according to their responsibilities
Responsible: elcuco, volunteers needed

Plugins

Divide et impera!
Splitting application into independent modules gives lots of advantages:

  • ease of maintenance
  • ease of concurrent development
  • extensibility
  • linear complexity growth

Tasks:
  • designing robust plugin interface
  • implementing plugin-related mechanics
  • managing third-party plugin contributions
Responsible: volunteers needed

Qt toolkit integration

Qt comes with a bunch of useful and well-done set of tools. It would be weird if DevQt would ignore this fact and provide no integration with them.
Tasks:

  • making use of Qt formats (i.e. documentation format)
  • implementing integration with Qt tools (Designer, Assistant, Linguist, qmake and others)
Responsible: Cesar, volunteers needed
