Instead of prologue

DevQt team are not young girls and boys in a sandbox anymore, playing developer games, proudly releasing new versions of not yet ready, buggy software, called DevQt. That's why it is suggested to choose the versioning scheme and decide how to make sure the next version is ready for release.

Assigning version numbers

As DevQt is tightly bound to Qt itself, so adopting Qt's versioning scheme looks reasonable.
This way each version will look like DevQt-X.Y.Z, where

  • X — MAJOR version number
  • Y — MINOR version number
  • Z — patch level

Releasing new version

DevQt exists for less then a month, but even now it has some unresolved bugs registered on BerliOS bugtraq. Then why are those new versions still released? To solve this issue a brand-new version creation cycle suggested:

  1. Defining TODO list for new version.
  2. Implementing the TODO list. No new bells and whistles should be added except those, noted in current TODO list.
  3. Team testing. Fixing bugs. Increase patch level, to see version number progress :)
  4. Announcing new version on QtCentre. Testing by QtCentre inhabitants. Fixing bugs. Increase patch level, to see version number progress :)
  5. Put some new screenshots on BerliOS screenshot page.
  6. Build ready to use packages and put them on BerliOS download section.

Deploying DevQt as ready to use packages

The target audience of DevQt is developer community. As a developer one should be able to compile it himself, but speed up getting started with DevQt for those, who cannot/doesn't want to do this for some reason, maintaining prebuilt packages might be handy.
